# Whitepaper: NeuralShell Security and Provenance Architecture

Version: draft v1  
Date: 2026-03-04

## Abstract

NeuralShell is a desktop AI workstation designed around controllable IPC boundaries, local-first operation, and verifiable release artifacts. This paper describes the architecture and threat model implemented in the repository.

## 1. Problem

General-purpose chat clients often optimize speed and convenience over deterministic trust boundaries. For sensitive workloads, operators need:

1. constrained process boundaries
2. auditable message flow
3. verifiable release artifacts
4. predictable failure behavior

## 2. Architectural Model

Core layers:

1. Electron main process:
   - trusted orchestration layer
   - IPC handlers and policy enforcement
2. Electron renderer:
   - sandboxed UI layer
   - no direct privileged Node access
3. Bridge/runtime services:
   - model routing
   - session and state persistence

## 3. Security Controls

Implemented control patterns include:

1. strict webPreferences:
   - `contextIsolation: true`
   - `nodeIntegration: false`
   - `sandbox: true`
2. IPC channel allowlisting and payload validation
3. release gate tests for structural regressions
4. checksum/provenance generation for release assets

## 4. LLM Execution Model

LLM calls use:

1. model + persona state management
2. streaming and non-streaming pathways
3. timeout, retry, and cancellation behavior
4. bridge health and autodetect diagnostics

## 5. Release Provenance Flow

Release evidence in this codebase is generated by:

1. `scripts/release-manifest.js`
2. `tear/release-status.js`
3. `scripts/release-checksums.js`
4. `tear/verify-release-freshness.js`

Expected outputs:

1. `release/manifest.json`
2. `release/status.json`
3. `release/provenance.json`
4. `release/checksums.txt`
5. `release/checksums.json`

## 6. Threat Model Summary

Mitigated classes:

1. renderer privilege escalation via IPC abuse
2. accidental release drift via gate failures
3. silent artifact tampering when checksums are used correctly

Residual risk classes:

1. compromised host machine
2. compromised dependency upstream before detection
3. operational misconfiguration outside repository controls

## 7. Verification Evidence

Current engineering evidence should be read from:

1. `MASTER_PROOF.md`
2. `HARD_PROOF_v1.0.0-OMEGA.md`
3. `artifacts/local-smoke-report.json`

## 8. Conclusion

NeuralShell demonstrates a practical approach to secure desktop AI operations by combining strict IPC design with verifiable release mechanics. The model is intentionally operational: claims are tied to scripts, tests, and generated artifacts.
